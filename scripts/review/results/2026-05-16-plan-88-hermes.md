# Adversarial plan review — issue #88 route-state validation

- Date: 2026-05-16
- Plan: `docs/plans/2026-05-16-issue-88-route-state-validation.md`
- Reviewer: Hermes delegated adversarial review
- Verdict: MAJOR

## Blocking findings

1. The plan did not guarantee prevention during normal offline report generation.
   - It kept generation offline-first and made live validation an explicit closeout command.
   - That could detect drift only if someone remembered to run the validator; generators could still emit closed issue targets when committed route maps drift.

2. The plan missed hardcoded closed fallback routes in weekly code.
   - `scripts/llm_wiki_weekly_freshness.py` lines 350-354 hardcode fallback routes to #76 and #79 when `data/issue_routing_map.json` is absent.
   - A data-only route-map fix would leave reachable closed-target defaults.

3. The planned tests were too narrow.
   - Existing tests assert current bad behavior: weekly recommendations include #76/#79; OSS MoorDyn routes to #79.
   - The plan needed tests for missing route-file defaults, slug-specific OSS closed routes, and generated output safety without a separate validator.

4. The design allowed “route maps and/or route helpers,” which permitted an insufficient data-only fix.
   - The plan must require both route-map correction and code-level safe normalization/defaults.

## Required revisions

- Require updates to weekly built-in defaults so no fallback route points to closed #76/#79.
- Require code-level safe routing in both weekly and OSS generators for active-update actions.
- Add tests for weekly missing-map fallback, OSS slug-specific closed-route fallback, and generator output safety with committed maps.
- Tighten acceptance criteria to mandatory code + data changes.
