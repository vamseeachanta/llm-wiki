# Issue #29 Source Title Aliasing Implementation Surface Intel

## 1. Existing assets

- Source corpus root is `wikis/marine-engineering/wiki/sources/`, not `knowledge/wikis/marine-engineering/wiki/sources/`. The issue-body prefix is stale for this spinout repository.
- Tracked marine source pages: `git ls-files 'wikis/marine-engineering/wiki/sources/*.md' | wc -l` returned **19,171**. The issue summary says about 19,159; `wikis/marine-engineering/wiki/index.md` frontmatter says `source_count: 19167`, so implementation should recompute rather than trust any one count.
- Representative filename-style pages:
  - `wikis/marine-engineering/wiki/sources/37161.md` has frontmatter title `37161.pdf`, slug `37161`, and metadata table fields `collection`, `filename`, `extension`, `size_bytes`, `year`, and `relative_path`.
  - `wikis/marine-engineering/wiki/sources/omae2010-20360.md` has title `OMAE2010-20360.pdf` with the same metadata shape.
  - `wikis/marine-engineering/wiki/sources/013.md` has title `013.PDF` with the same metadata shape.
- Existing index/search-adjacent code:
  - `scripts/llm_wiki_strengthening_scorecard.py` defines `page_title(...)`, `count_indexed_paths(...)`, `source_sort_key(...)`, `top_items(...)`, `make_portal(...)`, and `ensure_index_portal_link(...)`.
  - `wikis/marine-engineering/wiki/index.md` has a curated `## Sources` table followed by many generated filename-style rows such as numeric PDF labels.
  - `wikis/marine-engineering/wiki/portal.md` is generated by `scripts/llm_wiki_strengthening_scorecard.py` when `--write` is used.
- Existing safety/governance assets:
  - `scripts/scan_source_families_safe.py` and `tests/test_scan_source_families_safe.py` provide the pattern for public-safe metadata-only scanning.
  - `scripts/validate_completion_artifacts.py` and `scripts/validate_governance_artifacts.py` enforce public-safety scans on existing reports/governance docs.
  - `docs/plans/2026-05-11-issue-29-source-title-aliasing.md` already records the issue #29 planning boundary and proposed commands.
  - `docs/reports/2026-05-11-safe-source-family-aggregate-scan.md` documents aggregate-only raw-source scanning constraints.

## 2. Implementation surface

- Add `scripts/source_title_aliases.py`: deterministic alias generator for committed wiki source pages. It should parse frontmatter plus the local metadata table, never raw source files.
- Add tests, likely `tests/test_source_title_aliases.py`: fixture-driven unit tests for title extraction, alias preservation, ambiguity reporting, public-safety filtering, and idempotence.
- Change `scripts/llm_wiki_strengthening_scorecard.py`: update `page_title(...)` or a new display-title helper to prefer `canonical_title`/`display_title` while keeping existing link targets stable; update `source_sort_key(...)` to classify aliases consistently.
- Change `wikis/marine-engineering/wiki/index.md` generation path or add a chunk/index companion if issue #28 owns pagination. The source table should display readable titles where available without renaming `sources/*.md`.
- Add a policy doc such as `docs/source-title-aliasing-policy.md` if implementation needs durable evidence precedence and ambiguity rules.
- Add generated ambiguity output under `docs/reports/`, preferably aggregate/safe rows using source slugs and filenames only, not raw archive paths.

## 3. Deterministic metadata sources

- Available in committed source pages: YAML frontmatter (`title`, `slug`, `domain`, dates, tags) and a `## Metadata` table with `collection`, `filename`, `extension`, `size_bytes`, `year`, and often `relative_path`.
- Available in curated source pages: several hand-authored source rows in `wikis/marine-engineering/wiki/index.md` already have human-readable labels/descriptions and can guide display behavior.
- Negative finding: sampled autogenerated pages do not contain paper/report titles beyond filename-style title/heading and metadata table. `collection` plus `year` plus `filename` is usually not enough to infer a canonical human title.
- Negative finding: no implemented `scripts/source_title_aliases.py` or test file for source-title aliasing exists. Current alias terms appear in the issue #29 plan, not in implementation.
- Safety finding: source-page metadata includes raw absolute `path` fields. The aliasing implementation should not copy those into public reports or new indexes; use slug, filename, collection, year, and relative repo path instead.

## 4. TDD plan

- `test_preserves_raw_filename_alias`: every updated page keeps the original filename/title in `title_aliases`.
- `test_skips_opaque_numeric_without_title_evidence`: pages like `37161.pdf` or `013.PDF` remain ambiguous unless committed safe evidence provides a title.
- `test_uses_existing_readable_title_without_churn`: hand-authored readable source pages retain their title and get stable alias metadata without unnecessary rewrites.
- `test_writes_ambiguity_report_without_private_paths`: report includes slug/filename/status/counts, excludes raw absolute paths and private-root subpaths.
- `test_is_idempotent`: a second run produces no diff.
- `test_index_display_prefers_canonical_title_but_keeps_link_target`: index/search display changes labels only; links still point to the original `sources/<slug>.md`.
- `test_source_metadata_parser_handles_case_and_sparse_fields`: `PDF`/`pdf`, missing `year`, and missing `relative_path` are deterministic.

## 5. Risks and unknowns

- Path mismatch: issue body names `knowledge/wikis/...`; actual repo path is `wikis/...`.
- Count mismatch: observed tracked count is 19,171, issue summary is about 19,159, and index frontmatter says 19,167. Treat count as dynamic.
- Ambiguity is the dominant case for numeric conference/PDF filenames. Guessing titles from collection/year/track paths would create false labels.
- Volume/performance: 19k files is manageable, but full-repo `rg`/`git grep` over all wikis can be slow in this workspace. Implementation should stream files once and avoid repeated whole-repo scans.
- Raw metadata leak risk: existing source pages may contain raw absolute `path` metadata. New reports and indexes must avoid propagating those fields.
- Coordination risk with issue #28: if canonical source index chunking lands first, alias display should plug into the same display-title helper rather than duplicate index logic.

## 6. Public-safety notes

- Do not open raw private archives to infer titles.
- Do not OCR, summarize, or copy vendor standards text, client/project material, private manifests, figures, tables, or secrets.
- Use only committed source-page metadata and already-authored public-safe wiki text.
- Preserve stable links and raw filenames as aliases; do not rename source files as part of issue #29.
- Ambiguous pages should be reported as ambiguous rather than labeled speculatively.
