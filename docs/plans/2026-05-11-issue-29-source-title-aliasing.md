# Issue #29 Plan — Canonical Source-Title Aliasing

## Gate status

- GitHub issue: [vamseeachanta/llm-wiki#29](https://github.com/vamseeachanta/llm-wiki/issues/29)
- Current local plan state: `status:plan-approved`
- Implementation status: **approved for future implementation; not implemented in this exit-closeout pass**
- Scope class: metadata normalization/searchability work over already committed wiki source pages.

## Resource intelligence

Fresh local inspection on 2026-05-11 found:

- Actual repo source path: `wikis/marine-engineering/wiki/sources/`.
- Issue body path prefix `knowledge/wikis/...` is stale for this spinout repo; implementation must use `wikis/...`.
- Source-page count by filesystem: 19,171 committed markdown files.
- Many source pages currently use filename-style frontmatter and headings, for example `title: "37161.pdf"` and `# 37161.pdf`.
- Source pages contain a local metadata table with fields such as collection, filename, extension, size, year, and sometimes a relative path. These fields are not always sufficient to infer a human-readable paper/report title.
- Codex scouting found a negative signal that sampled autogenerated pages do not contain paper/report titles beyond filename-style headings plus sparse metadata; issue #29 must report ambiguous pages rather than guessing from collection/year/track clues.
- Codex scouting found a safety risk: existing source-page metadata may include raw absolute source paths, so new reports/indexes must emit slug, filename, collection, year, and repo-relative paths only.
- Existing generator/scoring surface: `scripts/llm_wiki_strengthening_scorecard.py` has `page_title(...)` and index parsing logic that can be extended or kept compatible with alias metadata.

## Public-safety boundary

Allowed:

- Use already committed source-page frontmatter/headings/metadata tables.
- Add public-safe alias fields such as `canonical_title`, `title_aliases`, `display_title`, and `alias_confidence` only when derived deterministically from committed metadata or public-safe evidence.
- Preserve raw filename as a stable alias.
- Generate aggregate ambiguity reports without path-rich private source locations.

Forbidden:

- Re-opening raw private-archive PDFs or documents to infer titles.
- Copying private path strings into new reports or generated indexes.
- Promoting vendor standards text/tables/figures, client material, CAD/model content, admin/personal data, secrets, or private manifests.
- Renaming source files in a way that breaks existing links.

## Implementation plan after approval

1. **Write failing tests first.**
   - Build small fixture source pages with filename-only, numeric-only, already-readable, and ambiguous title cases.
   - Assert raw filename preservation as an alias.
   - Assert deterministic aliases and ambiguity output.
   - Assert no raw private paths are emitted to public reports.

2. **Add a deterministic aliasing script.**
   - Proposed path: `scripts/source_title_aliases.py`.
   - Inputs: domain name and source directory.
   - Outputs:
     - updated source-page frontmatter for deterministically safe aliases, or a generated sidecar registry if frontmatter churn is too large;
     - ambiguity report under `docs/reports/` with counts and sample slugs, not private paths;
     - optional machine-readable JSON summary for validation.

3. **Define metadata precedence.**
   - Do not invent titles from opaque filenames.
   - Preserve existing readable titles when already non-filename-like.
   - If a committed source page has a safe descriptive field that is clearly a title, prefer it as `canonical_title`.
   - Otherwise keep filename title and add `alias_status: ambiguous`.
   - Always keep raw filename in `title_aliases`.

4. **Update index/search display surfaces.**
   - Extend index generation or display helper logic to prefer `canonical_title`/`display_title` where present.
   - Preserve target links and slugs.
   - Coordinate with issue #28 so chunked source-index pages also use readable display labels where available.

5. **Document alias policy.**
   - Proposed path: `docs/source-title-aliasing-policy.md`.
   - Include evidence hierarchy, ambiguity rules, public-safety restrictions, and regeneration commands.
